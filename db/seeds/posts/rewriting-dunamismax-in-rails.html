This site used to be an Astro static build. As of today, it is a Ruby on Rails 8 application. The pages, the projects list, the blog you are reading right now, and the RSS feed all come from Active Record models served by ERB templates. The frontend/ directory is gone, the Bun toolchain is gone, the Docker container is gone, and the static-output pipeline is gone with them.

The short version: I just shipped a Rails site for SentryPact, I liked the shape of it more than the Astro shape I had here, and I wanted my personal site to use the stack I am actually building products in. So this is the rewrite log.

Why move off Astro

Astro was a fine choice for this site when I made it. It produces clean static HTML, the islands model is honest about where JavaScript actually belongs, and the build is fast. None of that has changed.

What changed is what the site is for. dunamismax.com is supposed to be a long-running personal site that explains the work I do, hosts a blog I keep writing in, and links to projects I keep building. Almost every project in that list is now a Rails app. SentryPact is a Rails control plane. Scrybase, DebugPath, MyLifeRPG, and the chess platform are all converging on Rails 8 on SQLite for the small versions and Rails on PostgreSQL once they need it. Keeping my own site on a different stack from everything else I ship had stopped making sense.

The other thing was the content model. Astro's content collections are good, but they are Markdown files in a folder. That is fine, until you want a real list view, a real ordering, a real "featured" flag, or a real JSON or RSS feed driven from the same source of truth. Active Record gives me all of that with a single seed file and two tables.

What the stack is now

  • Ruby 4.0.3 on Rails 8.1.3
  • Hotwire (Turbo + Stimulus) over import maps
  • Propshaft for the asset pipeline
  • Tailwind CSS via tailwindcss-rails, layered on top of a small hand-written CSS sheet
  • SQLite for primary, cache, queue, and cable databases
  • Puma with Solid Queue running in-process
  • Caddy on the host for TLS and reverse proxy, then Cloudflare at the edge
  • systemd as the process supervisor, no containers in production

This is the same shape that runs sentrypact.com. One box, one Git checkout, one redeploy script, one set of operational commands. If something breaks on one site I already know how to debug it on the other.

Rails renders pages on request, Caddy on the host handles TLS and the cache layer, and Cloudflare proxies in front of it.

Routes and controllers

The URL surface stayed almost identical, on purpose. Old links should still work. 

GET  /          pages#home
GET  /about     pages#about
GET  /contact   pages#contact
GET  /projects  projects#index
GET  /blog      posts#index
GET  /blog/:id  posts#show
GET  /feed.xml  feeds#index
GET  /robots.txt feeds#robots
GET  /up        rails/health#show

PagesController is intentionally tiny — three actions, all of them static. ProjectsController#index groups projects by category. PostsController uses friendly slugs via to_param. FeedsController hands back an RSS feed and a robots.txt. The Rails health endpoint at /up is what Caddy and any uptime monitor hit.

Content as Active Record

The blog and the projects list are both database-backed now. Two models, two migrations, one seed file: 

class Post < ApplicationRecord
  scope :published, -> { where(published: true) }
  scope :recent,    -> { order(published_on: :desc, id: :desc) }
  def to_param = slug
end

class Project < ApplicationRecord
  scope :ordered,  -> { order(:position, :name) }
  scope :featured, -> { where(featured: true) }
end

Posts have a slug, title, description, body HTML, tag list, publish date, and a published flag. Projects have a slug, name, tagline, category, status, visibility, repo URL, optional product URL, stack list, position for ordering, and a featured flag.

Blog post bodies live as HTML files under db/seeds/posts/. The seed file reads them at db:seed time and writes them into the posts.body_html column. That gives me the lightest possible authoring story (open a file in an editor, paste HTML), keeps the source of truth in Git, and lets the database be the read path.

When I want a richer authoring story later, I can swap that for Action Text, for a Markdown renderer, or for a real CMS surface that writes to the same table. The Active Record schema is the contract; the seed pipeline is just one way to populate it.

Layout and styling

The layout is one ERB file, app/views/layouts/application.html.erb, with a shared header and footer partial. Tailwind is in via tailwindcss-rails for utility classes. A hand-written sheet at app/assets/stylesheets/application.css defines the design tokens: colors, radii, shadows, spacing, and the dark-and-light theme variables.

A single Stimulus controller flips the theme between dark and light and remembers the choice in localStorage. The page paints in the user's last theme on first load via a tiny inline script in the head, before any CSS arrives, so there is no light flash on a dark page.

This is the same approach the SentryPact site uses. The two sites share a family of CSS variables, a similar grid system, and the same theme controller pattern. They look distinct because the colors are different, not because the underlying mechanics are different.

Deployment

Deployment is the same shape that runs sentrypact.com:

  • The repo lives in /home/sawyer/github/dunamismax.com
  • Ruby and Node come from mise, pinned by .mise.toml
  • Gems are installed deployment-style under vendor/bundle
  • Production env lives in /etc/dunamismax-web/env
  • Puma is launched by a systemd unit and binds to 127.0.0.1:8082
  • Caddy on the host terminates TLS and reverse-proxies to Puma
  • Cloudflare proxies the public domain

Redeploy is a five-line script: git pull, bundle install, source the env file, bin/rails db:prepare, bin/rails assets:precompile, systemctl restart dunamismax-web. There is no container build, no image registry, no CI deploy job. The box pulls its own code.

Cloudflare keeps doing what it was already doing for the Astro site. The origin certificate is the same Cloudflare Origin CA cert in Full (strict) mode. The only change at the edge is that the origin now serves dynamic HTML from Rails instead of static HTML from a built Astro site.

What I gave up

A static-only site is faster than a Rails site at handing out the first byte of /. That difference is real, and I gave it up. In return I get a real data model behind the content, a real admin path I can grow into, background jobs available as soon as I want them, a console I can drop into, ActiveRecord migrations as the changelog for the site, and the same developer experience I am using on the products that live on this site.

The site is also no longer purely static, which means I need Caddy to be up, Puma to be up, and the SQLite databases to be on disk. That trade is worth it for what I want this site to do over the next few years.
