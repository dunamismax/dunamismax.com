The strongest argument for Rails is not nostalgia, syntax, or a framework checklist. It is the way a Rails app keeps the shape of a product small enough for one person to hold in their head.

A useful product is rarely just a controller and a template. It needs data modeling, authentication, background work, email, caching, assets, deployment, logs, tests, security checks, and a way to change all of that without losing the thread. Rails does not make those problems vanish. It puts them in familiar rooms with names on the doors.

One App, One Language

The modern default is often a split stack: API here, frontend there, jobs in another process, schema changes somewhere else, glue code in between. That can be the right call for a large team. For a small team, or for one person building a real product, every extra boundary becomes another place where context leaks.

Rails keeps the product close. A model migration, a background job, a mailer, a controller action, a view, and a test can live in the same repository and speak the same language. When the feature changes, the blast radius is easier to see.

The Schema Is The Spine

I like Rails because it treats the database as part of the application, not an implementation detail hiding behind a service boundary. Migrations become the product's history. Associations document the shape of the domain. Constraints and validations make the happy path less fictional.

This matters more than it sounds. If the data model is explicit, everything above it becomes easier to reason about. The UI has something real to show. Background jobs have a durable source of truth. Exports and admin surfaces are not special cases; they are different views over the same system.

Hotwire Is Enough More Often Than People Admit

A lot of web apps need fast navigation, optimistic-feeling updates, small interactions, and forms that do not punish the user. They do not always need a separate client application.

Hotwire sits in the useful middle. The server still owns rendering. Turbo makes normal links and forms feel fast. Stimulus handles the interaction that actually belongs in the browser. You can still write JavaScript when it earns its keep, but you do not start every feature by inventing a protocol between two apps you now have to maintain.

Small Operations Are A Feature

The deployment story matters. A Rails app that runs on one box with Puma, systemd, Caddy, and a boring database is not a compromise by default. It is a system with fewer moving parts, fewer accounts, fewer bills, and fewer hidden failure modes.

That does not mean one box forever. It means starting with an operating shape that is easy to inspect and recover. Add complexity when the product earns it. Until then, keep the path from code to running software short.

The Real Test

The real test of a stack is not whether it looks impressive in a diagram. It is whether you can come back six months later, understand the feature, change it safely, deploy it without ceremony, and sleep after it goes live.

Rails keeps passing that test for the kind of software I want to build: self-hostable, inspectable, privacy-aware products with clear data models and enough interface polish to feel good without turning every screen into a separate application.

That is why Rails is my default. Not because every problem is a Rails problem, but because many real products are smaller than the stacks people wrap around them.
