This is the build log for getting sentrypact.com online. The site itself is a Rails 8.1 app — the public marketing surface and the control plane that the future native clients will eventually sit on top of. The goal here was to get it served on a real domain, on real hardware, in a shape I can keep running for a long time without thinking about it.

The end state is intentionally boring:

  • Rails 8.1 + Puma, listening on a loopback-only port
  • SQLite on disk for primary, cache, queue, and cable databases
  • systemd as the process supervisor
  • Caddy in front of it terminating TLS and proxying back to localhost
  • One Ubuntu VM, one Git checkout, one git pull to redeploy

No platform layer, no managed Rails host, no separate database server, no Redis. Just Rails the way Rails is happy to run.

The Rails app

SentryPact is a monorepo. The web surface lives in apps/web. It is a stock Rails 8.1 app with the defaults turned on: Hotwire (Turbo + Stimulus) over import maps, Propshaft, Tailwind via tailwindcss-rails, Solid Queue, Solid Cache, and Solid Cable. Everything that used to require a Redis is now a SQLite database.

The four production SQLite databases live next to each other: 

apps/web/storage/production.sqlite3
apps/web/storage/production_cache.sqlite3
apps/web/storage/production_queue.sqlite3
apps/web/storage/production_cable.sqlite3

The point of using all four is that I do not need a separate cache, queue, or pubsub service to run the app. They are local files I can copy off the host with rsync if I have to. That is the whole disaster recovery story for v0.1, and that is fine for a pre-launch site.

The toolchain

Ruby 4.0.3 and Node 24.13.1 are pinned by .mise.toml and .ruby-version. mise manages both. The first install on the box looked like this: 

curl -fsSL https://mise.run | sh
echo 'eval "$(~/.local/bin/mise activate bash)"' >> ~/.bashrc
cd /path/to/sentrypact
mise trust
mise install

Building Ruby 4.0.3 from source needs the usual list of headers and toolchains. On Ubuntu, that means a one-time apt-get install of build-essential, libssl-dev, libreadline-dev, zlib1g-dev, libyaml-dev, libffi-dev, libgmp-dev, libsqlite3-dev, and friends. I do this once and then forget about it.

Gems are installed deployment-style under apps/web/vendor/bundle: 

cd apps/web
gem install bundler -v 4.0.11 --no-document
bundle config set --local deployment true
bundle config set --local without 'development test'
bundle config set --local path vendor/bundle
bundle config set --local frozen true
bundle install

This keeps gems off the system path, which means the box can host more than one Rails app later without their dependencies bleeding into each other.

Secrets and environment

Production env lives in a single file, /etc/sentrypact-web/env, owned root:sawyer with mode 0640. It carries SECRET_KEY_BASE, RAILS_ENV=production, PORT=8081, APP_HOST=sentrypact.com, RAILS_HOSTS, RAILS_SERVE_STATIC_FILES=1, WEB_CONCURRENCY=2, RAILS_MAX_THREADS=5, and SOLID_QUEUE_IN_PUMA=true.

The Solid Queue inside Puma flag is the concession to running on one box. Instead of a separate jobs process, the Puma plugin runs the Solid Queue supervisor in-process. For a single-server deployment that is the simplest shape that works.

SECRET_KEY_BASE was generated on the box with SECRET_KEY_BASE_DUMMY=1 RAILS_ENV=production bundle exec bin/rails secret and then pasted into the env file. The dummy variable on the generation step is the Rails 8 idiom that lets you boot enough of the app to print a secret without having one yet.

systemd

Puma runs under systemd. The unit lives at /etc/systemd/system/sentrypact-web.service. The interesting parts are:

  • EnvironmentFile=/etc/sentrypact-web/env so Rails sees the production env without anything baked into the unit
  • ExecStart uses the absolute path to the mise-installed Ruby's bundle so systemd does not depend on shell activation
  • BUNDLE_GEMFILE is set explicitly because systemd does not inherit the working-directory bundler magic
  • ProtectSystem=full, ProtectHome=read-only, PrivateTmp=true, and NoNewPrivileges=true are on
  • ReadWritePaths is whitelisted explicitly to storage, tmp, and log under the app directory

Puma binds to 127.0.0.1:8081. There is no public listener on the Rails process. The whole point of the systemd hardening is that even if something inside the app tries to scribble outside its directories, systemd refuses.

Boot looks like: 

sudo systemctl daemon-reload
sudo systemctl enable --now sentrypact-web.service
sudo systemctl status sentrypact-web.service

Caddy

Caddy is the only thing on the public network. It owns ports 80 and 443, handles TLS via Let's Encrypt automatically, and reverse-proxies to the Puma socket. The vhost is two blocks: www redirects to apex, apex proxies through. 

www.sentrypact.com {
    redir https://sentrypact.com{uri} permanent
}

sentrypact.com {
    encode zstd gzip

    header {
        X-Content-Type-Options "nosniff"
        X-Frame-Options "DENY"
        Strict-Transport-Security "max-age=31536000; includeSubDomains"
        Referrer-Policy "strict-origin-when-cross-origin"
        X-XSS-Protection "0"
        Permissions-Policy "camera=(), microphone=(), geolocation=()"
        -Server
    }

    reverse_proxy 127.0.0.1:8081 {
        header_up X-Forwarded-Proto https
        header_up X-Real-IP {remote_host}
    }
}

The Rails side trusts that header set because the production environment has config.assume_ssl = true and config.force_ssl = true, with /up exempted so the loopback health probe still works on plain HTTP.

After a config change Caddy gets caddy validate first, then systemctl reload caddy. reload, not restart — Caddy is happy to swap configs without dropping connections.

The redeploy loop

Once the host was set up, redeploys collapsed to a five-line shell script: 

cd /home/sawyer/github/sentrypact && git pull
cd apps/web
eval "$(~/.local/bin/mise activate bash)"
bundle install
set -a && . /etc/sentrypact-web/env && set +a
bundle exec bin/rails db:prepare
bundle exec bin/rails assets:precompile
sudo systemctl restart sentrypact-web

No Capistrano, no Kamal, no GitHub Actions deploying anything. The box pulls its own code, runs migrations against SQLite (effectively a no-op most of the time), precompiles assets through Propshaft, and bounces the service.

db:prepare is the right Rails command here, not db:migrate. It creates the database if it does not exist and runs pending migrations otherwise. On a fresh box the first deploy and every later deploy use the same command.

What I left out

Two things were tempting and I deliberately skipped them.

Kamal. The Rails 8 default deploy story is Kamal, and the kamal gem is in the Gemfile so it is available. The checked-in config/deploy.yml is still scaffold configuration. For one box with one app and one domain, Kamal is more moving parts than I want today. When SentryPact gets a second host, or has to coordinate the Rails control plane with a separate jobs host, Kamal becomes the better answer. Until then, systemd plus Caddy is fewer abstractions to debug.

PostgreSQL. The Solid trio (Queue, Cache, Cable) all run fine on SQLite, and the marketing site has no real write volume. When the control plane starts holding subscriptions, devices, pact state, and audit events, that is the cue to move primary off SQLite. The data layout here is designed to make that move boring, not to delay it forever.

Health checks

Three commands cover the day-to-day: 

curl -s http://127.0.0.1:8081/up                              # local Puma
curl -s --resolve sentrypact.com:443:127.0.0.1 \
     https://sentrypact.com/up                                # via Caddy
sudo journalctl -u sentrypact-web -f                          # tail logs

/up is the Rails-built health endpoint. It returns 200 when the process is running and the framework boots cleanly. That is what a load balancer or uptime monitor should hit. It is excluded from the SSL redirect and from host authorization so it works behind any front door.

What this stack is good for

The shape above is good for a small, real product that needs to keep running. The whole site is a directory I can tar, an env file I can scp, a systemd unit I can cat, and a Caddyfile I can read. There is no part of the deployment that I cannot explain in one sitting, and there is no part of the deployment that requires a vendor to be online for the site to be up.

That is the standard for the rest of the public work too. If a deployment needs a slide deck to explain, it is doing too much.
